## База даних

Схема `publishing` моделює видавничий бізнес і містить 8 таблиць:

| Таблиця | Призначення |
|---|---|
| `Authors` | Автори книг |
| `Employees` | Співробітники (редактори, перекладачі тощо) |
| `Books` | Каталог книг |
| `Orders` | Замовлення клієнтів |
| `AuthorBook` | Зв'язок M:N між авторами та книгами |
| `EmployeeBook` | Зв'язок M:N між співробітниками та книгами |
| `OrderItem` | Позиції замовлень (M:N Orders ↔ Books) |
| `Contracts` | Контракти авторів і співробітників |

### Практична 3 — Проєктування схеми (DDL)

**Файл:** `practical_3.sql`

**Тема:** Початкове проєктування реляційної схеми бази даних `publishing` на основі ER-діаграми.

**Зміст скрипту:**
- Визначення 8 таблиць із первинними ключами та типами даних
- Налаштування зовнішніх ключів через `ALTER TABLE ... ADD FOREIGN KEY`
- Таблиця `Contracts` використовує поле `PersonType` (`author` / `employee`) для поліморфного зв'язку
- Асоціативні таблиці `AuthorBook`, `EmployeeBook`, `OrderItem` із складеними первинними ключами

> **Примітка:** Це попередня версія схеми — вона відображає результат проєктування на етапі ER-моделювання. Фінальна, доопрацьована версія схеми з повними обмеженнями, CHECK-constraints та тестовими даними — у `practical_4.sql`.

**Порядок виконання:**

```
1. Відкрити practical_3.sql у Workbench
2. Переконатись, що база даних (або інша схема) обрана:
   CREATE DATABASE IF NOT EXISTS publishing_v3;
   USE publishing_v3;
3. Виконати весь скрипт (Ctrl+Shift+Enter)
```

> ⚠️ Скрипт не містить `CREATE DATABASE` та `USE` — виконувати на чистій схемі,
> щоб уникнути конфлікту з основною базою `publishing`.

### Практична 6 — Тригери

**Файл:** `practical_6.sql`

**Тема:** Складні SQL вирази — тригери для валідації бізнес-логіки.

**Задачі:**

| # | Зміст |
|---|---|
| 1 | Тригер `trg_contracts_bi` (BEFORE INSERT) — перевірка цілісності контракту |
| 2 | Тригер `trg_contracts_bu` (BEFORE UPDATE) — ті самі правила при оновленні |
| 3 | Тестові INSERT: 1 коректний + 3 навмисні помилки |
| 4 | SELECT-перевірка актуальних контрактів |

**Логіка тригерів (обидва):**
- Рівно один із `AuthorID` / `EmployeeID` має бути заповнений
- `ContractType` має відповідати власнику (`Author` або `Employee`)
- `EndDate` не може бути раніше `StartDate`

**Порядок виконання:**

> ⚠️ Задачу 3 запускати **по одному INSERT за раз** — Workbench зупиняється на першій помилці.

```
1. Виконати Задачу 1 (тригер BEFORE INSERT) — окремим блоком
2. Виконати Задачу 2 (тригер BEFORE UPDATE) — окремим блоком
3. Задача 3 — виконувати кожен INSERT окремо:
   a. Коректний INSERT → має виконатись без помилок
   b. Два власники → помилка: 'Exactly one of AuthorID or EmployeeID must be set'
   c. Невірний ContractType → помилка: 'ContractType must match owner'
   d. EndDate < StartDate → помилка: 'EndDate must be >= StartDate'
4. Виконати Задачу 4 (SELECT перевірка)
```

### Практична 7 — Вкладені запити та VIEW

**Файл:** `practical_7.sql`

**Тема:** Вкладені запити, CTE, віконні функції, повторне використання коду.

**Задачі:**

| # | Зміст |
|---|---|
| 1 | NOT EXISTS — автори, чиї книги жодного разу не замовляли |
| 2 | HAVING + підзапит — книги з продажами вище середнього |
| 3 | CTE + RANK() OVER (PARTITION BY) — рейтинг книг у межах жанру |
| 4 | CREATE OR REPLACE VIEW — повторне використання агрегованого запиту |

**Порядок виконання:**

```
Передумова: practical_4.sql вже виконано.

1. Задача 1: виконати весь блок (INSERT тестових даних + SELECT)
2. Задача 2: виконати SELECT з HAVING
3. Задача 3: виконати CTE-запит
4. Задача 4: спочатку CREATE OR REPLACE VIEW, потім SELECT з VIEW
```

> **Примітка:** Задача 1 додає тестового автора (`Тестовий Автор`) і книгу без замовлень — це необхідно для демонстрації NOT EXISTS.

**Порядок виконання:**

```

Задача 5 містить ALTER TABLE — для чистого логу:
```
